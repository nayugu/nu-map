ISpecialTerms work terms

Defines every type of non-course block the institution supports — co-ops, internships, exchanges, research terms, or any semester that displaces coursework. Leave types empty and the work-experience section in BankPanel disappears entirely.

Port key
"specialTerms"
Consumed by
BankPanel (drag sources), SemRow/SummerRow (grid rendering), GradPanel (via computeGrantedAttrs), PlannerContext (drop validation, span maps), planModel (PDF export)
Override when
Your institution has structured work experiences, exchanges, or leave semesters that occupy planner grid slots

The duration vs weight distinction

Every duration option has two numeric fields. Understanding this distinction is critical:

FieldWhat it isWhere it's used
durationLength in calendar monthsStored in plan state, used for display labels and drag-drop identity
weightGrid weight on the ICalendar scale (1.0 = one full semester)Span arithmetic: termWeight > semSlotWeight → spans into next slot

Contract — SpecialTermType fields

FieldTypePurpose
idstringMachine-readable type identifier, e.g. "coop", "intern". Stored as typeId in plan state. The planner iterates types generically — no type ID is hardcoded in core logic.
labelstringDisplay name shown in BankPanel, grid chips, and PDF export.
colorstringCSS hex color for grid chips and BankPanel headers.
durationsSpecialTermDuration[]Available duration options. Each entry is draggable from BankPanel.
attributeGrantsstring[]Attribute codes automatically satisfied when any term of this type is placed. e.g. NU co-op grants ["EX"]. Empty array for no grants.
occupiesSlotbooleanTrue if this term displaces regular coursework. False for part-time or non-credit experiences.
creditValuenumberCredits awarded. 0 for unpaid co-ops; non-zero for credit-bearing exchanges or research terms.

Contract — SpecialTermDuration fields

FieldTypePurpose
idstringMachine-readable duration identifier, e.g. "4mo", "6mo". Used as part of the term's drag identity.
labelstringDisplay label in BankPanel and grid chips, e.g. "4-month", "6-month".
durationnumberCalendar months. Stored in plan state. Used by resolveTermByDuration() to find the right descriptor from a saved plan.
weightnumberGrid weight on the same scale as ICalendar.SemesterType.weight. 1.0 = occupies one full-semester slot. Controls span logic via termSpans(termWeight, semSlotWeight).

Generic default

// src/adapters/generic/specialTerms.js
export default { types: [] };

Empty array → BankPanel's work-experience section is hidden. No drag sources, no grid slots for special terms.

Implementing for your institution

Add one entry to types for each distinct kind of non-course block your institution supports:

// Example: semester exchange program
{
  id:              "exchange",
  label:           "Semester Exchange",
  color:           "#60a5fa",     // blue-400
  durations: [
    { id: "1sem", label: "1-semester", duration: 5, weight: 1.0 }
  ],
  attributeGrants: ["IC"],        // exchange satisfies Interpreting Culture
  occupiesSlot:    true,
  creditValue:     16,
}
✦
Weight must match ICalendar scale. If your institution uses fall/spring at weight 1.0 and a term occupies exactly one full semester, use weight 1.0. If it spans a full year, use 2.0. If it only uses a summer half-session, use 0.5.

Attribute grants: The attributeGrants array is how you encode "a placed co-op counts as EX" without any hardcoded string in the core. computeGrantedAttrs() in specialTermUtils.js unions these arrays across all placed terms and passes the result to getCoverage().

How span logic works end-to-end

flowchart LR A["User drops 6-month co-op\nonto Fall 2026 slot\n(sem weight: 1.0)"] B["PlannerContext stores\nspecialTermPl[termId] =\n{ typeId:'coop', duration:6, semId:'fall-2026' }"] C["resolveTermByDuration(durations, 6)\n→ { weight: 2.0 }"] D["termSpans(2.0, 1.0)\n→ true"] E["specialTermStartMap['fall-2026'] = termId\nspecialTermContMap['spring-2027'] = termId"] F["SemRow renders Fall 2026 as start block\nSemRow renders Spring 2027 as continuation block"] A --> B --> C --> D --> E --> F
The weight fields drive everything. No type ID is checked in the span logic — it's purely arithmetic.
