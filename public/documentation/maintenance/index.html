nu-map/documentation / maintenance

Maintenance operational playbook

nu-map has a live course database that Northeastern updates each semester. There is no structured API — data accuracy depends on manual scraping plus manual correction. All data updates are initiated by you; nothing runs automatically. In practice, every maintenance task is triggered by exactly one of two events: it is the start of a semester (periodic data refresh), or you find bad data. Know which trigger you're responding to, follow its path, and one push to main deploys everything.

Trigger A — periodic data refresh (start of semester)

There is no automated scraper. At the start of each semester (January, May, September) you manually run Catalog Check and NUPath Update to discover what has changed, then apply and push. This is the most common maintenance action — most of the time it takes under 30 minutes.

Creating a patch to override a catalog value

Use a patch when our data is correct and the catalog is wrong (typo, stale entry). Patches are re-applied on every deploy — they survive future scraper runs.

  1. Open Dev Portal → Patches tab.
  2. Select Update, enter the course code, fill in only the wrong field(s).
  3. Click Generate YAML. Copy output to data/patches/YYYY-MM-DD_description.yaml.
  4. Locally: npm run data:patch:write && npm run build — verify in Preview tab.
  5. Commit and push. The patch takes precedence over whatever the scraper writes.
# data/patches/2025-09-01_fix-cs3700-credits.yaml
meta:
  date: "2025-09-01"
  author: "developer"
  reason: "Credits corrected per Fall 2025 catalog"
courses:
  update:
    - subject: CS
      number: "3700"
      fields:
        credits: 4
        nuPath: ["AD", "ND"]

Trigger B — you find bad data

A user reports a wrong title. A course shows the wrong NUPath. A whole subject's credits look off. The right response depends on scale: a single known field → patch it directly; unknown scope or bulk problem → use the dev portal's Work tab to investigate and apply.

Choose the right tool

SituationToolWhy
One course, you know exactly which fieldPatches tabFastest. No scrape. Patch survives all future deploys.
Unknown scope — want to see what differs across all coursesCatalog CheckFull field comparison: title, credits, description, prereqs, scheduleType. Shows exactly what changed.
NUPath attributes specifically seem wrongNUPath UpdateFetches from Tableau — more authoritative than the catalog HTML for attribute data. Per-course selective apply.
New course not in database (you know the code)Patches tab → AddInstant. For a batch of new courses, use Catalog Check instead.
Course discontinued, should be removedCatalog Check → ❌ filter → Apply FixesDeletes record. Use git checkout -- public/all-courses.json to undo if wrong.

Catalog Check path — full decision tree

Use this when you need to verify scope before touching anything. Runs locally only — requires npm run dev. See Catalog Check reference for how the tool works internally.

NUPath Update path

Use instead of Catalog Check when NUPath attributes are the specific issue. Manual mode gives you per-course review; Automatic mode commits directly to main on a schedule. See NUPath Update reference for mechanics.

flowchart LR NP["Work tab → NUPath Update\nManual mode → Scan for Changes"] NP --> RES["Inspector: + Gained / − Lost / ± Mixed\nAll courses pre-selected by default"] RES --> UNCHECK["Uncheck courses you don't trust\n(suspicious change, verify on Tableau first)"] UNCHECK --> APPLY["☢ Apply Selected Fixes\nwrites nuPath field only\ndoes not touch other fields"] APPLY --> PUSH_NP["npm run build → Preview → push"]

One push deploys everything

Both trigger paths end here. Pushing to main — whether by merging a PR or pushing directly — triggers the same automated pipeline. You never deploy manually. Patches are re-applied automatically, so a patch committed weeks ago still protects against a bad scraper value today.

Data pipeline — where data comes from

Understanding the pipeline helps you interpret what you see in the scraper PR or Catalog Check results.

flowchart LR subgraph External ["External sources"] SN["SearchNEU / nu-courses\nterm availability, section data"] CAT["catalog.northeastern.edu\ntitles, descriptions, credits, prereqs"] TAB["tableau.northeastern.edu\nNUPath attributes (authoritative)"] GN["sandboxnu/graduatenu\nmajor requirement JSON trees"] end subgraph Build ["Build pipeline"] FETCH["npm run data:fetch:write\nbase course records from SearchNEU"] SCRAPE["npm run data:scrape:write\noverlays richer descriptions from catalog"] PATCH["npm run data:patch:write\napplies all data/patches/*.yaml"] JSON["public/all-courses.json\n(~6.5 MB, ~10k entries)"] MAJORS["public/majors/\n(~1,471 JSON files)"] end SN -->|manual: npm run data:fetch:write| FETCH CAT -->|optional manual step| SCRAPE TAB -->|NUPath Update tool| PATCH GN -->|manual sync| MAJORS FETCH --> SCRAPE --> PATCH --> JSON

Key insight: all-courses.json is a built artifact — do not hand-edit it. Edit source data (patches, scripts) and let the pipeline regenerate it. All data refresh steps are manual; data:patch:write is the only step that runs automatically (as part of every CI build). Run data:scrape:write manually when catalog descriptions need refreshing.

Forked repos — when they need attention

graduatenu (major requirement trees)

public/majors/ comes from sandboxnu/graduatenu. It needs manual syncing when Northeastern adds a new catalog year or the upstream fixes requirement bugs.

  1. Copy the new year's files from packages/api/src/major/majors/YEAR/ into public/majors/YEAR/.
  2. Run node scripts/fix-pooled-requirements.js — fixes a parser bug in ~297 files where elective pools are structured as checkbox lists instead of credit-hour pools.
  3. Update src/data/majorLoader.js to include the new year in the dropdown.
  4. npm run build → verify in Preview → PR.

Known upstream parser issues (all handled in nu-map, no action needed unless you sync):

IssueAffectedFix
Elective pools shown as checkboxes instead of credit pool~297 files (2024/2025)fix-pooled-requirements.js post-processor
OR-wrapped sections ("Supporting Courses") show "One of (0/8)"Various majorsRuntime normalization in allocateSection() — no JSON changes needed
Missing creditHoursRequired on General Electives sections~386 filesDefaults to 0 if absent — known gap

nu-courses (term availability)

The ninest/nu-courses fork provides which semesters each course actually runs in. Updated by the upstream maintainer each semester. nu-map pulls from it via the SearchNEU API wrapper in scripts/fetch.js — no manual sync needed unless the API endpoint changes.

End-of-semester checklist

Run through this at the start of each semester (January, May, September):

  • Open Dev Portal locally → Changes tab — scan for any unexpected bulk changes since last semester.
  • Work tab → Catalog Check → Launch Check — review all discrepancy categories, apply the ones that look correct.
  • Work tab → NUPath Update → Manual scan — apply attribute updates from Tableau.
  • Create patch files for any known catalog errors (catalog wrong, our data correct).
  • If a new catalog year is available upstream in graduatenu, sync public/majors/ and run fix-pooled-requirements.js.
  • npm run build locally → verify in Preview tab across iPhone, iPad, Desktop.
  • Work tab → Review & Push → push to main — CI deploys in ~2 minutes.

Environment setup (first time)

git clone https://github.com/nayugu/nu-map.git
cd nu-map
npm install
npm run dev    # Vite on :5173, dev portal backend on :3333

Node 18+ required. No environment variables needed — the dev portal password is a PBKDF2 hash embedded in the client HTML. The portal performs no privileged server operations.
