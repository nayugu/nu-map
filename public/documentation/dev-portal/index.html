Dev Portal public/dev.html

The dev portal is a single self-contained HTML file (~2,400 lines) that gives you visibility into the course database everywhere, and write access to it locally. That distinction — read-only on GitHub Pages, full write access on localhost — is the one thing that determines everything about how the portal behaves. Start there.

Remote URL
https://nayugu.github.io/nu-map/dev.html
Local URL
http://localhost:5173/dev.html (after npm run dev)
Write operations talk to
http://localhost:3333 — started by npm run dev

Local vs. remote — the capability split

The first thing the page does on load is ["localhost","127.0.0.1",""].includes(location.hostname). This single boolean controls almost every UI decision:

flowchart LR subgraph Local["localhost — full access"] direction TB L1["Auth skipped entirely"] L2["Work tab: Catalog Check + NUPath Update + Push to GitHub"] L3["Changes tab shows ⏳ Pending push badge\nfor entries not yet on GitHub Pages"] L4["Header: 'LOCAL' badge + dev_logo.png"] L5["Preview iframe points at local build"] L6["Link: 'Open on GitHub Pages →'"] end subgraph Remote["GitHub Pages — read-only"] direction TB R1["Auth required (PBKDF2 hash check)"] R2["Work tab hidden\n(HTTPS → HTTP mixed-content blocked)"] R3["Overview shows 'Open Local →' hint + port input"] R4["Header: logo.png"] R5["Preview iframe points at deployed build"] end

Why is the Work tab blocked on GitHub Pages? The write tools communicate with http://localhost:3333. A page served over HTTPS cannot make requests to an HTTP endpoint — browsers block this as mixed content. The portal detects this at load time and hides the Work tab rather than letting you click buttons that silently fail.

Read-only tabs — available everywhere

Overview

The landing tab after login. Contains: a PR review reminder (links to GitHub PRs), a data status card (course count + last-updated timestamp from data-meta.json), repository quick-links, and on GitHub Pages only — the "work tools are local-only" hint card with a copyable npm run dev command.

Changes tab

A paginated audit log of everything that has happened to the database. Reads change-log.json (and scrape-state.json for rotation status). On localhost it also fetches the deployed changelog from GitHub Pages to compute which local entries haven't been pushed yet — those get a ⏳ Pending push badge.

Patches tab

A form-driven YAML generator. No server call — pure client-side string building. Fill in the course code and the fields to change, click Generate YAML, copy the output to data/patches/YYYY-MM-DD_description.yaml. Three action modes:

ActionRequired fieldsWhat it generates
Addsubject, number, title, creditscourses.add[] entry
Updatesubject, number + at least one changed fieldcourses.update[].fields
Removesubject, numbercourses.remove[] entry

The download button auto-names the file YYYY-MM-DD_action-course.yaml. Patches are re-applied on every deploy — committing a patch means it persists through all future scraper updates.

Data Sources tab

A reference card showing where all course data comes from and how each source is updated. Contains two cards:

  • Course Data Sources table — four rows: catalog.northeastern.edu (titles, descriptions, credits, prereqs), tableau.northeastern.edu (NUPath attributes, authoritative), ninest/nu-courses via SearchNEU (term availability, live sections), sandboxnu/graduatenu (major requirement trees). Each row shows how that source is updated.
  • Update Workflow — five numbered steps for a full data refresh: Catalog Check → NUPath Update → apply patches → build → Review & Push.

This tab is informational only — no server calls, available on both localhost and GitHub Pages.

Write tools — local only (Work tab)

These are the tools that actually modify all-courses.json on disk. Both require npm run dev to be running. Their internal mechanics — SSE streams, discrepancy categories, apply pipelines — are documented in the sub-pages below. This section covers what they are and when to reach for each.

Push to GitHub

A "Review & Push" button that shows a git diff before committing. A two-step confirmation prevents accidental pushes. Under the hood:

sequenceDiagram participant You participant Portal as Browser participant Server as localhost:3333 You->>Portal: click "Review & Push" Portal->>Server: GET /git-status Server-->>Portal: {dirty, status (file list), diff (text)} alt nothing to commit Portal-->>You: "✓ Nothing to commit" — button disabled else files changed Portal-->>You: modal shows changed files + diff You->>Portal: "Confirm & Push" Portal->>Server: POST /git-push Server->>Server: git commit && git push origin main Server-->>Portal: {ok, msg} Portal-->>You: result banner — GitHub Actions picks up in ~1 min end

The iframe is clipped to the device's safe area — the notch and home bar regions are covered by a white #pv-screen-bg div and the actual bezel border, accurately simulating what iOS renders. Reset state clears localStorage key ncp-state-v2 inside the iframe — useful for testing first-run onboarding.

Auth system (remote only)

The password is never stored in plain text. A developer runs node scripts/generate-dev-password.js which outputs a PBKDF2-SHA256 hash embedded in the page. At login, the browser hashes the entered password with 300,000 iterations and compares hex strings — all in Web Crypto, no server involved. On success, sessionStorage (not localStorage) stores the session token — it clears when the tab closes, so a shared machine can't be left permanently unlocked.
