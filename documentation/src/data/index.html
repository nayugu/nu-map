data/ I/O layer

Three thin I/O modules. No business logic. Each wraps one type of external resource: major/minor requirement JSON files and browser localStorage. Course fetching and normalization belong entirely to the ICourseCatalog adapter — there is no courseLoader module.

Parent
src/
Called by
PlannerContext.jsx (exclusively)
Rule
I/O only — no business logic, no React, no institution imports

Children

concept
course loading
Course loading lives in the ICourseCatalog adapter (courseCatalog.fetchAll()). No separate module exists — the adapter owns fetch and normalize.
module
majorLoader.js + minorLoader.js
Async-fetches Major2 requirement JSON from the graduatenu fork. Provides grouped dropdown options for the major/minor selector.
module
persistence.js
Reads and writes the main plan state blob to localStorage. Four functions: storageKey, loadSaved, saveState, clearState — all take storagePrefix as a parameter.

Full data pipeline

sequenceDiagram participant App as App startup participant PS as persistence.js participant CC as courseCatalog adapter participant LS as localStorage participant FS as public/ (static files) participant API as live API (fallback) participant ML as majorLoader.js participant GN as graduatenu/ fork App->>PS: loadSaved(storagePrefix) PS->>LS: read {prefix}-state-v2 LS-->>PS: saved JSON or null PS-->>App: hydrated state (or defaults) App->>CC: courseCatalog.fetchAll() CC->>FS: GET /northeastern/all-courses.json alt local file ok FS-->>CC: 6.5 MB JSON array else local file fails CC->>API: GET husker.vercel.app/courses/all API-->>CC: JSON array end CC-->>App: Course[] (already normalized) Note over App: User selects major in GradPanel App->>ML: loadMajor(path) ML->>GN: GET parsed.initial.json GN-->>ML: Major2 requirement tree ML-->>App: major JSON Note over App: Any state mutation App->>PS: saveState(prefix, persist, planObj) PS->>LS: write {prefix}-state-v2
Complete data flow. Course loading is handled by the adapter — fetchAll() owns both fetch strategy and normalization.
