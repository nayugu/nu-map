context/ State layer

Three React contexts that bridge adapter data, application state, and UI. InstitutionContext injects the adapter bundle; PlannerContext owns all application state; ThemeContext manages CSS variable injection.

Reads theme from localStorage, injects 100+ CSS variables onto <html>. Cycle between dark/light themes.

InstitutionContext feeds adapter data directly to components that need it. PlannerContext is the state hub for everything else.

Provider nesting order

The three providers must nest in this exact order in App.jsx. InstitutionProvider is outermost because PlannerContext calls useInstitution() internally during initialization.

// src/App.jsx — required nesting order
<InstitutionProvider adapter={institutionAdapter}>
  <ThemeProvider>
    <PlannerProvider>
      <PlannerApp />
    </PlannerProvider>
  </ThemeProvider>
</InstitutionProvider>

The no-prop-drilling contract

No component passes state down via props (except trivial configuration like semId to a row). All shared state comes from usePlanner(). Institution config comes from usePort(IPortName). This makes it easy to add features — any component can read or mutate any state without threading props through the tree.

The trade-off: adding new state means editing PlannerContext.jsx, which is already large. When adding state, check first whether it belongs in the context (shared across components) or as local component state (only needed in one place).
