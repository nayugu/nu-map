Catalog Check Work tab · local only

Catalog Check scrapes catalog.northeastern.edu subject by subject and compares every field in all-courses.json against what the catalog says today. It streams results live so you can act before the full run finishes. The key skill is reading the four discrepancy categories — they have different priorities and different consequences when you apply fixes.

Scan endpoint
GET http://localhost:3333/run (Server-Sent Events stream)
Fix endpoint
POST http://localhost:3333/fix (streaming response)
Requires
npm run dev — starts Vite + the port 3333 server together
Blocked on
GitHub Pages (HTTPS → HTTP mixed-content; tab is hidden entirely)

The check is a live stream — stoppable at any time

When you click Launch Check, the browser opens an EventSource connection to localhost:3333/run. The server scrapes one subject at a time and sends a subject_done event after each one. The stat cards, progress bar, and live log all update per-subject in real time — you do not have to wait for the full run.

Four discrepancy categories — different meaning, different priority

Each subject_done event feeds ckAccumulate(d), which appends items to the module-level ckDiscrepancies[] array. Every item has one of four types. The type determines both how urgently to act and what the fix does:

BadgetypeWhat it meansPriorityApply Fixes effect
⚠ Need PRchangedA real field changed — title, description, credits, NUPath codes, prereqs, scheduleType. Every diff in this item has isRangeOnly: false.HighOverwrites the changed fields in all-courses.json
≈ Range diffrangeOnly numeric range fields differ (e.g. min/max credits shifted slightly). Every diff has isRangeOnly: true.Low — usually ignoreOverwrites the range fields only
🆕 New coursesaddedIn the catalog but not in our database. Server attaches catalogData to the item so it can be inserted.High — verify firstInserts full course record from catalogData
❌ Not in catalogmissingIn our database but the scraper found no match. May be discontinued, renamed, or a transient catalog outage.Medium — investigate before actingDeletes the course record — irreversible without git

How discrepancies accumulate in the Inspector

flowchart LR EVT["subject_done event\n{changes[], added_courses[], missing_courses[]}"] EVT --> ACC["ckAccumulate(d)\n\nfor each {code, diffs} in changes:\n allRange = every diff.isRangeOnly?\n → push {code, type:'range'|'changed', diffs}\n\nfor each added_courses item:\n → push {code, type:'added', catalogData}\n\nfor each code in missing_courses:\n → push {code, type:'missing'}"] ACC --> ARR["ckDiscrepancies[]\n(grows throughout the run)"] ARR --> RENDER["ckRender()\n apply type filter\n apply search term\n cap display at 500\n update table + ck-fix-count label"]

The Inspector renders live — you can filter and search while the check is still running. Export JSON downloads the full ckDiscrepancies[] array as catalog-discrepancies-YYYY-MM-DD.json for offline analysis.

Apply Fixes — targeted mutation with a confirmation gate

The filter you have active when you click ☢ Apply Fixes determines exactly what gets sent to the server. This lets you be surgical — apply only the new courses without touching the field changes, for example.

Active filterWhat gets sent to POST /fix
AllEvery entry in ckDiscrepancies[]
⚠ ChangedOnly type === 'changed'
≈ RangeOnly type === 'range'
🆕 NewOnly type === 'added'
❌ MissingOnly type === 'missing'

Irreversibility: The only undo is git checkout -- public/all-courses.json. There is no in-portal rollback. After applying, run npm run build, verify in the Preview tab, then push to main.
